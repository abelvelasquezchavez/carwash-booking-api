# 🚀 Deployment Guide

This guide covers deploying the **Car Wash Booking API** to
[Render](https://render.com) with a [TiDB Cloud](https://tidbcloud.com)
(MySQL-compatible) database over SSL.

- **App host:** Render (Node web service, defined by [`render.yaml`](./render.yaml))
- **Database:** TiDB Cloud Serverless (MySQL 8 wire-compatible, port `4000`, SSL required)
- **Migrations:** `prisma migrate deploy` runs during the Render build

---

## 0. Prerequisite — commit the initial migration ⚠️

`prisma migrate deploy` **only applies migrations that already exist** under
`prisma/migrations/`. It never generates them. If that folder is empty, the
Render build will succeed but **create no tables**.

So, **before the first deploy**, generate the initial migration locally and
commit it:

```bash
# Against your local MySQL (docker compose up -d mysql)
npm run prisma:migrate -- --name init     # = prisma migrate dev --name init
git add prisma/migrations
git commit -m "chore(db): add initial migration"
git push
```

Migrations authored against MySQL are compatible with TiDB Cloud (TiDB speaks
the MySQL protocol). From then on, every push triggers a Render build that runs
`prisma migrate deploy` and applies any pending migrations to TiDB.

## 1. Create the TiDB Cloud database

1. Create a **Serverless** cluster in TiDB Cloud.
2. Create a database (e.g. `carwash`).
3. Open **Connect** → choose **Prisma** (or "General") and copy the parameters:
   host, port (`4000`), user (looks like `xxxxxxxxxxx.root`) and password.

---

## 2. Build the `DATABASE_URL` (TiDB Cloud + SSL)

TiDB Cloud **requires TLS**. The exact format Prisma expects is:

```
mysql://<USER>:<PASSWORD>@<HOST>:4000/<DATABASE>?sslaccept=strict
```

Concrete example (note port `4000`, the `.root` user, and `sslaccept=strict`):

```
mysql://3xAmpLe1234.root:YOUR_PASSWORD@gateway01.us-east-1.prod.aws.tidbcloud.com:4000/carwash?sslaccept=strict
```

Key points:

- **Port is `4000`**, not `3306`.
- **`sslaccept=strict`** turns on TLS *and* verifies the server certificate
  against the trust store. This is the recommended, secure setting.
- If your **password contains special characters** (`@ : / ? # & %` …),
  URL-encode them. For example `p@ss:word` → `p%40ss%3Aword`.
- Append extra params with `&`, e.g. `...?sslaccept=strict&connection_limit=5`.

### Do I need the CA certificate?

In most environments you **do not**. TiDB Cloud Serverless presents a
certificate signed by a publicly trusted CA, and Render's runtime
(Debian-based) ships those root CAs in its system trust store at
`/etc/ssl/certs/ca-certificates.crt`. With `sslaccept=strict`, Prisma uses that
system store and the connection just works.

You only need to provide a CA file explicitly if you hit an error such as
`unable to get local issuer certificate` or `self-signed certificate in
certificate chain` — i.e. the host's trust store doesn't contain the CA.

### If Prisma *does* require the CA — `sslca`

1. Download the CA bundle from TiDB Cloud (**Connect** dialog → CA cert), or use
   your platform's known TiDB CA path.
2. Point Prisma at it with the **`sslca`** query parameter (path to a PEM file):

   ```
   mysql://<USER>:<PASSWORD>@<HOST>:4000/<DATABASE>?sslaccept=strict&sslca=/etc/secrets/tidb-ca.pem
   ```

   Relevant Prisma MySQL SSL params:
   | Param         | Meaning                                              |
   | ------------- | ---------------------------------------------------- |
   | `sslaccept`   | `strict` (verify cert) or `accept_invalid_certs`.    |
   | `sslca`       | Path to the CA certificate (PEM).                    |
   | `sslcert`     | Path to a client certificate (only for mTLS).        |
   | `sslidentity` | Path to a PKCS#12 client identity (only for mTLS).   |

3. **On Render**, ship the PEM via a **Secret File** (Dashboard → service →
   *Environment* → *Secret Files*):
   - Filename / mount path: `/etc/secrets/tidb-ca.pem`
   - Contents: paste the CA PEM.

   Render mounts secret files under `/etc/secrets/<filename>`, so the `sslca`
   path above lines up. Commit nothing — the cert stays a secret.

> Never use `sslaccept=accept_invalid_certs` in production — it disables
> certificate verification and exposes you to MITM attacks. Prefer fixing the CA
> trust with `sslca`.

## 3. Deploy to Render via the Blueprint

The repo ships a [`render.yaml`](./render.yaml) Blueprint.

1. In Render: **New** → **Blueprint** → connect this GitHub repo. Render reads
   `render.yaml` and proposes the `carwash-booking-api` web service.
2. Set the environment variables it can't generate:
   - **`DATABASE_URL`** → the TiDB Cloud URL from step 2 (marked `sync: false`,
     so it must be entered in the dashboard — never commit it).
   - `JWT_SECRET` is auto-generated by Render (`generateValue: true`).
   - `JWT_EXPIRES_IN`, `BUSINESS_OPEN`, `BUSINESS_CLOSE`, `NODE_ENV` come from
     the blueprint; adjust if needed.
   - `PORT` is injected by Render automatically — the app reads `process.env.PORT`.
3. Click **Apply**. The build runs:

   ```
   npm ci && npm run prisma:generate && npm run build && npx prisma migrate deploy
   ```

   `prisma migrate deploy` applies the committed migrations to TiDB (idempotent
   — safe to run on every deploy), then `npm start` boots the compiled server.
4. Render polls `GET /api/health`; once it returns `200` the service is live.

### (Optional) seed production data

Seeding is **not** part of the automated build (you rarely want to reseed prod).
Run it once, on demand, from the Render **Shell** tab:

```bash
npm run prisma:seed
```

This creates the admin `abuelo@carwash.com` / `carwash123` — **change that
password** (or remove the seed user) for a real production deployment.

---

## 4. Keeping the free instance awake (uptime cron)

Render's **free** web services spin down after ~15 minutes of inactivity, adding
a cold-start delay to the next request. To keep it warm, ping the health
endpoint on a schedule with any uptime service (e.g. UptimeRobot, cron-job.org):

```
GET https://<your-service>.onrender.com/api/health   every 10 minutes
```

The endpoint is public and cheap, so it's a safe keep-alive target.

## Environment variables (production)

| Variable         | Source on Render        | Notes                                   |
| ---------------- | ----------------------- | --------------------------------------- |
| `DATABASE_URL`   | Dashboard (`sync:false`)| TiDB Cloud URL + SSL (step 2).          |
| `JWT_SECRET`     | Auto-generated          | `generateValue: true` in blueprint.     |
| `JWT_EXPIRES_IN` | Blueprint (`1d`)        | Token lifetime.                         |
| `BUSINESS_OPEN`  | Blueprint (`8`)         | Availability opening hour.              |
| `BUSINESS_CLOSE` | Blueprint (`18`)        | Availability closing hour.              |
| `NODE_ENV`       | Blueprint (`production`)|                                         |
| `PORT`           | Injected by Render      | Do not set manually.                    |

## Troubleshooting

| Symptom                                            | Likely cause / fix                                                        |
| -------------------------------------------------- | ------------------------------------------------------------------------- |
| Build OK but every query 500s / "table not found"  | No migrations committed — run step 0, push, redeploy.                     |
| `unable to get local issuer certificate`           | Provide the CA via `sslca=/etc/secrets/tidb-ca.pem` (step 2).             |
| `Error: P1001: can't reach database server`        | Wrong host/port (`4000`), or TiDB cluster paused/IP not allowed.          |
| `Access denied for user`                           | Wrong user/password, or special chars not URL-encoded.                    |
| App boots locally but crashes on Render at startup | Missing `DATABASE_URL`/`JWT_SECRET` — env validation exits the process.   |
